# SmartCode 最终架构框架图

> 本文档用最直观的方式展示 SmartCode 项目的完整架构和审查系统的设计。
> 目标是**让完全不了解这个项目的人也能看懂**。

---

## 一、项目是什么

**SmartCode** 是一个终端 AI 编码 Agent——你给它一个任务（"帮我写一个登录模块"），它会自动：
1. 理解你的需求
2. 调用 AI 模型（如 Claude、GPT）生成方案
3. 执行工具（读文件、写代码、跑命令）
4. 反复迭代直到任务完成

本项目是 SmartCode 的 Python 版学习 Fork，核心代码在 `minicode/` 目录下，约 42K 行。

## 三、core loop 三阶段（最核心的概念）

整个 Agent 的工作流程就是一个 **"思考→行动→验证"** 的循环，理解这个就能理解整个项目。

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   用户输入 ──────────────────────────────────────────────────┐    │
│                                                              ▼    │
│  ┌───────────────── Prelude（准备阶段）──────────────────────┐    │
│  │                                                          │    │
│  │  ① 创建回合状态（TurnRecurrentState）                      │    │
│  │  ② 解析用户意图 → 构建 TaskObject                         │    │
│  │  ③ 检查上下文 token 用量 → 必要时压缩                      │    │
│  │                                                          │    │
│  └────────────────────────┬─────────────────────────────────┘    │
│                           ▼                                      │
│  ┌───────── Recurrent Kernel（循环核，一直重复直到完成）─────────┐│
│  │                                                              ││
│  │  Step A → 策略推导（这一步该探索/执行/验证？）                │││
│  │  Step B → 调 AI 模型（model_caller.py）                      │││
│  │  Step C → 判断返回（progress/retry/fallback/final）          │││
│  │  Step D → 执行工具（tool_executor.py）                       │││
│  │          ├─ write_file → 钩子 1（宽松审查）→ 钩子 2（写后）    │││
│  │          ├─ read_file → 直接执行                             │││
│  │          └─ run_command → 权限检查后执行                      │││
│  │              ↑                                               │││
│  │  没完成 ←───┘                                               │││
│  │                                                              ││
│  └────────────────────────┬─────────────────────────────────┘│
│                           ▼                                     │
│  ┌────────── Coda（收尾阶段）─────────────────────────────────┐  │
│  │                                                          │  │
│  │  ① 钩子 3：注入审查发现 + 沉淀到 skill/记忆                │  │
│  │  ② 记录调用统计（token 用量、耗时）                        │  │
│  │  ③ 更新任务状态（完成/暂停/失败）                           │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                           │                                      │
│                           ▼                                      │
│   返回消息列表给调用方（main.py / TUI）                         │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 这里的 Step D 最关键——它也是审查系统的插入点

```
Step D 展开：

模型返回 "用 write_file 写一个登录函数"
          │
          ▼
    ┌─────────────────────────────────────┐
    │  _execute_single_tool()             │
    │                                     │
    │  tool_name = "write_file"           │
    │                                     │
    │  ┌── 钩子 1：on_before_write() ──┐  │
    │  │  检查内容是否包含：             │  │
    │  │  ├─ 硬编码密钥？→ 阻断！       │  │
    │  │  ├─ SQL 注入？ → 阻断！        │  │
    │  │  ├─ eval()？  → 阻断！         │  │
    │  │  └─ TODO 遗留？→ 提示          │  │
    │  │  force=true 跳过阻断           │  │
    │  └──────────┬───────────────────┘  │
    │             │ 放行                  │
    │             ▼                       │
    │  tools.execute("write_file", ...)   │
    │             │                       │
    │             ▼                       │
    │  ┌── 钩子 2：on_file_written() ──┐  │
    │  │  ① 后台更新 import map        │  │
    │  │  ② 严格模式 → 检查是否触发     │  │
    │  │     审查子 Agent               │  │
    │  │     ├─ review: 跨文件分析      │  │
    │  │     └─ test: 沙盒跑测试        │  │
    │  └───────────────────────────────┘  │
    │                                     │
    │  返回 ToolResult                    │
    └─────────────────────────────────────┘
```

## 四、审查系统 3 个钩子详解

### 【钩子 1】写前宽松审查——安检门

**位置**：`tool_executor.py` → `_execute_single_tool()` → 工具执行前

**流程**：
```
用户写代码
    │
    ▼
提取文件内容 + 路径
    │
    ▼
白名单检查（test_/example_/mock_ 前缀跳过）
    │
    ▼
正则匹配
    ├── (api_key|secret|password|token) = "xxx"  → critical → 阻断
    ├── eval(                                  → major   → 阻断  
    └── #TODO / #FIXME                          → minor   → 提示
    │
    ▼
AST 检查
    └── bare except:  ← 捕获所有异常           → major   → 阻断
    │
    ▼
判断结果
    ├── 有 critical/major → 返回 ToolResult(ok=False)  ← 阻断写入
    │     用户加 force=true 可跳过（同时记录 false_positive）
    ├── 只有 minor        → 返回 ToolResult(ok=True) + 提示
    └── 无问题            → 返回 None（不干预，继续执行）
```

### 【钩子 2】写后处理——持续追踪

**位置**：`tool_executor.py` → `_execute_single_tool()` → 工具执行成功后

**流程**：
```
文件写成功
    │
    ├── → 后台线程更新 import map（不阻塞主循环！）
    │      ├─ AST 解析刚写的文件，提取所有符号（函数、类、变量）
    │      ├─ 遍历项目所有文件，找到每个符号的引用
    │      └─ 写入 .mini-code-import-map/import-map.json
    │
    └── → 如果是 strict 模式，检查是否触发严格审查
           │
           满足任一条件：
           ├── 改了 auth/login/security/payment 等安全路径
           ├── 改了 >200 行 或 跨 >10 文件
           ├── 这个文件 90 天内问题率 >60%
           └── 代码作者是新人（不在已知贡献者列表）
                │
                ▼
           启动审查子 Agent（agent_type="review"）
           ├── 工具集：只读（read_file / grep_files / find_references / ...）
           ├── 不准写文件，不准跑命令
           ├── 流程：
           │   1. 读 import-map.json → 找出受影响文件
           │   2. 读受影响文件 → 做兼容性分析
           │   3. code_review → 代码质量
           │   4. 输出结构化报告
           └── 报告存入 _coda_findings，等 Coda 阶段注入
           
           ╔═════════════════════════════════════════════════╗
           ║  审查子 Agent 是只读的，绝对安全！              ║
           ║  它不能写文件、不能删文件、不能跑命令             ║
           ║  报告注入到消息列表，主 Agent 自主决定是否采纳   ║
           ╚═════════════════════════════════════════════════╝
```

### 【钩子 3】Coda 阶段收尾——沉淀知识

**位置**：`loop_orchestrator.py` → `run_agent_turn()` → finally 块末尾

**流程**：
```
本轮循环结束
    │
    ▼
检查 _coda_findings 是否有内容
    │
    ├── 有审查报告 → 注入到消息列表
    │     {"role": "system", "content": "[Auto Review] ..."}
    │     主 Agent 下一轮会看到这份报告
    │     （是注入不是阻断，Agent 可以自己决定改不改）
    │
    └── 沉淀重要发现 → promote_findings()
           │
           判断规则：
           ├── critical 安全发现（密钥/SQL）→ skill
           │    写入 ~/.opencode/skills/<rule_id>.md
           │    跨项目生效，以后所有项目都自动防范
           │
           ├── 同一个规则本轮出现 ≥3 次 → skill
           │    高频犯错 → 自动变成系统提示
           │
           ├── 跨项目通用模式（os.path→pathlib 等）→ skill
           │
           ├── 架构类规则（arch- 开头）→ 全局记忆
           │
           ├── false_positive → 跳过
           │
           └── style/info 提示 → 跳过
```

## 五、两条子 Agent 链路

### 5.1 review 子 Agent——跨文件审查

```
什么时候触发：
  strict 模式下，钩子 2 检测到安全路径/大 diff/高问题率/新人代码 → 自动触发

工具集（全是只读的！）：
  read_file / grep_files / file_tree / find_symbols / find_references /
  diff_viewer / code_review

提示词（模型看到的指令）：
  "你是代码审查员。流程：
   1. 读 import-map.json 找出受影响文件
   2. 读受影响文件，做兼容性检查
   3. 对改动的文件运行 code_review
   4. 输出带严重级别的结构化报告
   你**只准读**文件，不准修改任何东西，不准跑测试。"

max_turns：8（最多 8 步，超时自动结束）
```

### 5.2 test 子 Agent——沙盒验证

```
什么时候触发：
  主 Agent 看到审查报告后，自主决定调用 task(agent_type="test")
  （不是自动触发，是 Agent 自己的判断——"审查报告说没问题，但我想确认一下"）

工具集：
  run_command / test_runner / read_file / list_files / git

流程：
  1. git worktree add --detach /tmp/sandbox HEAD  ← 创建沙盒
  2. cp 变更的文件到沙盒
  3. test_runner(path="/tmp/sandbox")  ← 跑测试
  4. 全部通过 → 报告 [TEST_RESULT: PASS]
  5. 有失败  → git checkout HEAD <file> 只回滚失败文件
               ⚠️ 新建文件用 rm -f 回滚（git checkout 不处理未跟踪文件）
               报告 [TEST_RESULT: FAIL] + 错误详情

测试失败后的处理链路：
  ① 失败报告注入消息列表，LLM 下一轮看到
  ② 沉淀到 review_memory（.mini-code-import-map/review-findings.json）
  ③ 同规则失败 ≥3 次 → 自动沉淀为 skill，下次写代码时自动防范
```

## 六、import map 机制

### 为什么需要 import map

AI 改了一个函数时，需要知道"这个函数被哪些文件调用了"来检查兼容性。
不用 `grep`（太慢且跨平台不一致），而是**提前建一个索引表**。

### 数据结构

```json
{
  "version": 1,
  "updated_at": 1716979200.0,
  "symbols": {
    "authenticate": {
      "file": "auth/login.py",
      "type": "function",
      "is_public": true,
      "referenced_by": ["views.py", "api.py"]
    }
  }
}
```

### 生命周期

```
首次启动（背景线程，不阻塞）：
  build_import_map()  →  遍历所有 .py 文件  →  AST 提取每个文件的符号
  →  对每个符号搜索引用  →  写入 .mini-code-import-map/import-map.json
  →  耗时 3-8 秒（视项目大小）

每次写文件后（钩子 2，后台线程）：
  update_import_map_for_file()  →  AST 解析新文件提取符号
  →  遍历项目所有文件找引用  →  更新 JSON
  →  不阻塞主循环！Coda 阶段才等待线程完成

审查时（O(1) 查表）：
  review agent 读 import-map.json  →  找到受影响文件列表
  →  只读这些文件做分析
  →  不用重新遍历整个项目
```

### 性能数据

| 操作 | 复杂度 | 耗时 | 是否阻塞 |
|:-----|:------:|:----:|:--------:|
| 首次建表 | O(n) | 3-8 秒 | 否（后台线程） |
| 增量更新 | O(n) | 2-5 秒 | 否（后台线程） |
| 查表 | O(1) | <1 毫秒 | 是（即时） |

## 八、运行时模式切换（新增能力）

> **核心问题**：如果启动时是 `MINICODE_REVIEW_MODE=off`，跑了一段时间写了代码，然后切到 `loose` / `strict`——
> 之前写过的代码会不会有漏网之鱼？
>
> **答案**：不会。系统检测到切换后会自动做一次全面回补扫描。

### 8.1 切换机制

```python
# 方式 1：环境变量（立即生效，不需重启）
export MINICODE_REVIEW_MODE=strict

# 方式 2：CLI 命令（TODO，后续实现）
/review-mode strict
```

每个钩子入口处都会执行模式切换检测：

```
on_before_write() / on_turn_end()
         │
         ▼
    get_review_mode()  ← 每次调用读环境变量，不是缓存的
         │
         ▼
    比较 _prev_mode（上次记录的模式）
         │
         ├── off → loose/strict → 触发全面回补扫描
         │                        + 启动 import map 建表
         │
         └── 其他切换（strict↔loose）→ 只记录日志，不扫描
```

### 8.2 全面回补扫描流程

```
检测到 off → strict/loose
         │
         ▼
   后台线程启动（不阻塞主循环！）
         │
         ├── 1. 全量建 import map（如果还没建）
         │     遍历项目所有 .py → AST 提取符号 → 搜索引用 → 写 JSON
         │
         ├── 2. 全量扫描所有 .py 文件
         │     对每个文件运行 _pre_review_content()
         │     ├── 正则检查：密钥硬编码？SQL 注入？eval？
         │     ├── AST 检查：bare except？
         │     └── 正则检查：TODO 遗留？
         │
         ├── 3. 汇总报告
         │     "扫描 42 个文件，发现 3 个严重问题，5 个建议"
         │     注入到 _coda_findings，Coda 阶段推送给主 Agent
         │
         └── 4. 沉淀严重问题
             critical 发现 → promote_findings() → skill / 全局记忆
```

### 8.3 三种场景对比

| 场景 | 行为 | 数据安全 |
|:-----|:-----|:--------:|
| 启动就是 loose/strict | 正常惰性建表，步步审查 | ✅ 全程覆盖 |
| off→strict 中途切换 | 触发回补扫描，事后补审所有已有代码 | ✅ 无漏网 |
| strict↔loose 互相切 | 只记录日志，不需要回补 | ✅ 已有覆盖 |

### 8.4 设计要点

```
为什么 `get_review_hooks()` 在 off 模式也返回实例？
  因为如果返回 None，切到 strict 后没有实例来检测过渡。
  实例始终存在，只是 off 模式下所有钩子都是 no-op（空操作）。

为什么回补扫描是后台线程？
  因为扫描全部 .py 文件可能耗时 3-10 秒。
  后台执行 + Coda 阶段注入，不阻塞主 Agent 的当前操作。

回补扫描会扫描已经写过的所有文件吗？
  是的。遍历项目根目录下所有 **/*.py 文件。
  对每个文件运行 _pre_review_content()。
  发现的严重问题会同时沉淀到 skill/全局记忆。
```

## 十一、面试话术（直接背诵）

> "我给 SmartCode 设计了一套完全插拔的两级审查系统。
>
> 整个 `review/` 目录存在即启用，删除即回归原版，`agent_loop.py` 只加了 3 处钩子共 20 行，全部用 `try/except ImportError` 兜底。
>
> **宽松审查**在 write 前执行正则+AST 检查，毫秒级完成，密钥泄露直接阻断写入。有 `force=True` 逃生口和 false_positive 自动学习——同一个 pattern 误报一次后不再拦截。`_fp_cache` 做内存缓存避免每次读盘。
>
> **严格审查**在改了 auth 安全路径、大规模 diff、历史问题率高的文件时自动触发。它不依赖主 Agent 的 LLM 决定，直接 `tools.execute("task")` 调审查子 Agent。子 Agent 只有只读工具，基于 import map（纯 Python 跨平台实现，首次启动时后台建表不阻塞）查受影响文件。写后增量更新也放后台线程，不阻塞主循环。
>
> import map 记录每个符号被哪些文件引用，审查时查表而非 grep 全项目，从 O(n) 降到 O(1)。
>
> 我做了三组对比实验：`off` vs `loose` vs `strict`，记录了阻断次数、问题发现率、误报率三组指标。"
