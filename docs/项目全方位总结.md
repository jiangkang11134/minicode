## 一、项目定位

SmartCode 是一个**终端 AI 编码 Agent**（类似 Claude Code 的开源 Python 实现）。用户通过命令行或 TUI 交互界面给 Agent 下达任务（如"帮我写一个登录模块"），Agent 会自主调用 AI 模型理解需求、执行工具读写代码、反复迭代直到完成。

| 属性 | 值 |
|:-----|:----|
| 语言 | Python 3.10+ |
| 运行时依赖 | **零**（纯标准库） |
| 源码量 | ~42K 行，91 个 .py 文件 |
| 测试量 | ~11K 行，31 个测试文件 |
| 入口 | `pip install -e .` 后 `minicode` 命令 |
| 模式 | TUI（交互终端）/ CLI（管道模式）/ 无头（headless） |

## 二、核心框架：`run_agent_turn()` 三阶段循环

这是整个项目的**心脏**。所有子系统都围绕这个循环组织。

```
入口文件：agent_loop.py（2 行，从 loop_orchestrator.py 重新导出）
实现文件：loop_orchestrator.py（878 行）
子模块：  model_caller.py（280 行，LLM 调用）
          tool_executor.py（159 行，工具执行 + 审查钩子 1/2）
决策引擎：turn_kernel.py（~1700 行，状态机）
```

### 关键设计点

- **状态机 vs 执行器分离**：`turn_kernel.py` 做"判断"（策略推导、响应分类、工具决策），`loop_orchestrator.py` 做"动作"（调模型、跑工具）。互不耦合。
- **四类响应**：不是简单的"好/坏"二分，而是 progress/retry/fallback/final 四类，每类有独立的处理路径。
- **Widening 机制**：深度模式卡住时自动切到广度模式，带 trigger_reason 和 evidence 的完整日志。
- **三阶段自动切换**：explore（先读）→ execute（再写）→ verify（验证），但不是死板的三步，是循环中动态推导的。

## 三、审查系统（review agent + test agent）

这是本次改造的核心新增。一套**完全插拔的两级审查系统**，`review/` 目录存在即启用，删除即回归原版。

### 3.1 文件结构

```
minicode/
├── review/                         ★ 新目录，删目录即卸载
│   ├── __init__.py
│   ├── config.py                   ★ 配置：MINICODE_REVIEW_MODE=off|loose|strict
│   ├── hooks.py                    ★ 3 个钩子 + _pre_review_content（唯一实现）
│   ├── mode_engine.py              ★ 4 层严格触发判断
│   ├── memory.py                   ★ 审查发现持久化（JSON 文件）
│   └── promotion.py                ★ 发现→skill/记忆 自动沉淀
│
├── tools/
│   ├── import_map.py               ★ NEW：AST 建表 + 增量更新
│   ├── sandbox_test.py             ★ NEW：Docker 沙箱测试执行器
│   ├── task.py                     ★ MOD：AGENT_TYPES 追加 review + test
│   └── code_review.py              ★ MOD：+1 行 re-export
│
├── tool_executor.py                ★ MOD：钩子 1（写前）+ 钩子 2（写后）
└── loop_orchestrator.py            ★ MOD：init_review_hooks + 钩子 3（Coda）
```

### 3.2 三级模式

| 模式 | 环境变量 | import map | 宽松审查 | 严格审查 | 子 Agent |
|:-----|:---------|:----------:|:--------:|:--------:|:--------:|
| L0: off | `MINICODE_REVIEW_MODE=off` | ❌ | ❌ | ❌ | ❌ |
| L1: loose | `MINICODE_REVIEW_MODE=loose` | ✅ 后台建 | ✅ 阻断 | ❌ | ❌ |
| L2: strict | `MINICODE_REVIEW_MODE=strict` | ✅ 后台建 | ✅ 阻断 | ✅ 四层判断 | ✅ review+test |

支持**运行时切换**：`export MINICODE_REVIEW_MODE=strict` 后下一轮钩子调用自动生效，
检测到 off→strict 过渡时自动触发**全面回补扫描**（建表 + 全量审查已有代码）。

### 3.3 宽松审查（钩子 1）

**位置**：`tool_executor.py` → `_execute_single_tool()` 工具执行前

**逻辑**：

```
on_before_write(tool_name, tool_input)
  │
  ├── 模式切换检测（off→strict 过渡时触发回补扫描）
  ├── 提取 content + file_path
  ├── _pre_review_content(content, file_path)
  │     ├── 白名单检查（test_/example_/mock_ 前缀跳过）
  │     ├── 正则匹配 _CRITICAL_PATTERNS：
  │     │   ├── (api_key|secret|password|token) = "xxx"  → critical
  │     │   ├── execute/exec(raw(...))                    → critical
  │     │   └── eval(...)                                 → major
  │     ├── 正则匹配 _MINOR_PATTERNS：
  │     │   └── #TODO / #FIXME / #HACK / #XXX             → minor
  │     └── AST 检查 bare except（捕获所有异常）           → major
  │
  ├── 有 critical/major → 阻断（返回 ToolResult(ok=False)）
  │     force=true 可跳过，同时记录 false_positive
  ├── 只有 minor → 放行但附带提示
  └── 无问题 → 返回 None（不干预）
```

### 3.4 严格审查（钩子 2 触发）

**位置**：`tool_executor.py` → `_execute_single_tool()` 工具执行成功后

**触发条件**（4 层判断，满足任一即触发）：

```
should_trigger_strict(file_path, ...)
  ├── 1. 安全路径：auth/ login/ security/ payment/ api/ middleware/ ...
  ├── 2. diff 特征：改 >200 行、跨 >10 文件、public API 变更
  ├── 3. 历史问题率：90 天内同一文件问题率 >60%
  └── 4. 新人代码：git author 不在已知贡献者列表
```

**触发后**启动两条子 Agent 链路：

```
钩子 2
  │
  ├── _spawn_review_sub_agent()
  │   └── tools.execute("task", agent_type="review")
  │        ├── 工具集：只读（read_file/grep_files/find_references/code_review/...）
  │        ├── 不准写文件，不准跑命令
  │        ├── 流程：
  │        │   1. 读 import-map.json → 找出受影响文件
  │        │   2. 读受影响文件 → 做兼容性分析
  │        │   3. code_review → 代码质量检查
  │        │   4. 输出结构化报告
  │        └── 报告存入 _coda_findings
  │
  └── _spawn_test_sub_agent()
      └── tools.execute("task", agent_type="test")
           ├── 工具集：{sandbox_test, read_file, grep_files, list_files}
           ├── 流程：
           │   1. 调用 sandbox_test(changed_files=[...])
           │   2. Docker 沙箱内跑 pytest
           │   3. 返回 [SANDBOX_RESULT: PASS] 或 [SANDBOX_RESULT: FAIL]
           └── 结果存入 _coda_findings
```

### 3.5 Docker 沙箱测试（sandbox_test 工具）

替代了原有的 `git worktree` + `git checkout` 方案，提供真正的容器级隔离。

```
sandbox_test(changed_files=["auth/login.py"])
  │
  └── DockerSandbox.run()
        ├── 1. docker info 检查 Docker 可用性
        ├── 2. docker run -d --rm python:3.11-slim（只读挂载项目目录）
        ├── 3. pip install -e /workspace（安装项目依赖）
        ├── 4. docker cp 变更文件进容器（覆盖只读挂载，使其可写）
        ├── 5. pytest -x --tb=short -q（300 秒超时）
        │     ├── returncode == 0 → PASS
        │     └── returncode != 0 → FAIL + 错误输出
        ├── 6. docker rm -f（销毁容器，零残留）
        └── 返回 [SANDBOX_RESULT: PASS] 或 [SANDBOX_RESULT: FAIL]
```

| 对比 | 旧方案（git worktree） | 新方案（Docker） |
|:-----|:----------------------|:----------------|
| 隔离性 | ❌ 共享文件系统 | ✅ 完全隔离 |
| 回滚 | `git checkout HEAD <file>` → 可能失败 | 删容器 = 回滚 |
| 新文件处理 | `git checkout` 报错，需要 `\|\| rm -f` | Docker 自动处理 |
| 跨平台 | Windows 上不稳定 | Docker Desktop 统一 |
| 用户需要 | git | Docker |

### 3.6 import map 机制

纯 Python 实现的跨平台符号索引，不依赖 grep/ripgrep。

```
首次启动（后台线程，不阻塞）：
  build_import_map() → 遍历所有 .py → AST 提取符号 → 搜索引用 → 写 JSON
  → 耗时 3-8 秒

每次写文件后（钩子 2，后台线程）：
  update_import_map_for_file() → 删除旧符号 → 提取新符号 → 搜索引用 → 更新 JSON
  → 不阻塞主循环，Coda 阶段 join(timeout=5) 等待完成

审查时（O(1) 查表）：
  get_affected_files("auth/login.py") → {"authenticate": ["views.py", "api.py"]}
  → 直接读 JSON，不需要全项目遍历
```

数据结构：
```json
{
  "version": 1,
  "updated_at": 1716979200.0,
  "symbols": {
    "authenticate": {
      "file": "auth/login.py",
      "type": "function",
      "is_public": true,
      "referenced_by": ["views.py", "api.py"]
    }
  }
}
```

### 3.7 审查结果沉淀（钩子 3 Coda 阶段）

```
on_turn_end() → _coda_findings 遍历 → 注入消息列表

同时 promote_findings()：
  ├── critical 安全发现（密钥/SQL）→ skill（~/.opencode/skills/）
  ├── 同规则本轮出现 ≥3 次 → skill
  ├── 跨项目通用模式（os.path→pathlib 等）→ skill
  ├── 架构规则（arch- 开头）→ 全局记忆（MemoryScope.USER）
  ├── false_positive → 跳过
  └── style/info → 跳过
```

### 3.8 运行时模式切换

```
启动时 MINICODE_REVIEW_MODE=off → 写了一些代码
          ↓
export MINICODE_REVIEW_MODE=strict
          ↓
下一轮钩子调用自动检测到过渡
          ↓

  ├── 全量建 import map（如果还没建）
  ├── 遍历项目所有 .py 文件
  ├── 对每个文件运行 _pre_review_content()
  ├── 汇总报告 → Coda 阶段注入
  └── 严重问题 → promote_findings()
```

## 四、Session 系统（会话持久化）

**文件**：`session.py`（约 600 行）

```
核心数据结构：
  SessionMetadata     → 会话列表元数据（id/时间/消息数/检查点数/...）
  FileCheckpoint     → 文件修改前的快照（用于回退）
  SessionData        → 完整会话状态（消息/转录/权限/技能/检查点/...）

核心操作：
  save_session(session, force_full=False)
    ├── 增量保存：仅追加新消息/新转录（字段级脏标记）
    ├── 每 10 次增量后触发一次全量保存
    ├── 写入 SESSIONS_DIR / <session_id> / deltas / <timestamp>.json
    └── 定期合并 delta（超过 50 个文件时强制合并）

  load_session(session_id)
    ├── 读取全量基础文件
    ├── 按时间序应用所有 delta 文件
    └── 恢复完整的 SessionData

  create_file_checkpoint(file_path) → FileCheckpoint
    ├── 写文件前调用，保存文件当前内容的快照
    └── 用于 /rewind 命令回退文件修改

  rewind_session(session_id, steps=1)
    ├── 按 step 数回退（找到最近的 N 个检查点）
    ├── 按 checkpoint_id 回退
    ├── 反向安全快照：回退前再保存一次当前状态
    └── 文件级原子回退

存储路径：
  ~/.mini-code/sessions/
    ├── index.json           ← 所有会话的元数据索引
    ├── <session_id>/
    │   ├── session.json     ← 全量数据
    │   └── deltas/
    │       ├── 00001.json   ← 增量 1
    │       ├── 00002.json   ← 增量 2
    │       └── ...
    └── ...
```

### 会话生命周期

```
用户输入
  → 主循环处理
  → 每次 write_file/edit_file 前 → create_file_checkpoint() 保存快照
  → 每 30 秒自动保存一次（增量）
  → 用户 /rewind → rewind_session() 文件回退
```

## 五、Memory 系统（跨会话记忆）

**文件**：`memory.py`（约 2500 行）

```
三个作用域：
  MemoryScope.USER     → ~/.mini-code/memory/     跨项目共享
  MemoryScope.PROJECT  → .mini-code-memory/       项目级（可版本控制）
  MemoryScope.LOCAL    → .mini-code-memory-local/ 本地（不提交）

四个层级（Atkinson-Shiffrin 模型）：
  WORKING   → 当前会话的工作记忆
  SHORT_TERM → 短期记忆（最近的项目知识）
  LONG_TERM  → 长期记忆（持久化）
  ARCHIVAL  → 归档（低优先级，仅显式搜索）

核心类：
  MemoryEntry           → 单条记忆条目（id/content/scope/tier/tags/category/...）
  MemoryFile            → 一个记忆文件（管理条目的增删改查+持久化）
  MemoryManager         → 全局管理器（搜索/注入/分类/自动分层）

搜索算法（BM25 + 多维评分）：
  对输入分词后：
  ├── BM25 相关性评分（词频 + 逆文档频率）
  ├── 子串匹配加分（部分匹配关键词）
  ├── 领域分类匹配（memory domain 过滤）
  ├── 标签匹配加分（精确标签命中）
  ├── 时效性加分（越新越靠前，指数衰减）
  ├── 使用频率加分（历史命中次数越多越靠前）
  └── 综合排序 → 返回 Top-N 条记忆

注入时机：
  build_system_prompt() 时从 MemoryManager 检索相关记忆
  → 按 scope 组装 → 注入到系统提示的 project_context 段
```

## 八、已修复的架构缺陷

| # | 缺陷 | 严重度 | 修复 |
|:-:|:-----|:------:|:-----|
| 1 | 钩子延迟初始化缺 tools | 🔴 | `get_review_hooks(cwd, tools, tool_context)` 三参数，在 `run_agent_turn()` 内 `init_review_hooks()` 初始化 |
| 2 | `_pre_review_content` 每次读盘 | 🟡 | `_is_false_positive` 优先查 `_fp_cache`（内存缓存），命中直接返回 |
| 3 | 增量更新阻塞主循环 | 🟡 | 更新放入 `threading.Thread`，`on_turn_end` 时 `join(timeout=5)` 等待 |
| 4 | `on_file_written` 入参不匹配 | 🟡 | 签名统一为 `(file_path, ...)`，调用方从 `tool_input` 提取路径 |
| 5 | `git checkout HEAD` 不处理新文件 | 🟡 | 改为 Docker 沙箱，容器销毁即回滚，不再需要 `git checkout` |
| 6 | 判断依据过时（lite→loop） | 🟢 | 全部更新为 `agent_loop.py` / `tool_executor.py` / `loop_orchestrator.py` |
| 7 | `_fp_cache` 并发竞争 | 🟢 | 加 `_fp_cache_lock: threading.Lock()` |
| 8 | off→strict 切换后已有代码漏审 | 🔴 | `get_review_hooks()` 始终返回实例；`_check_transition()` 检测过渡后自动触发全面回补扫描 |
| 9 | git worktree 沙箱隔离性差 | 🟡 | 替换为 `sandbox_test` Docker 容器沙箱，完全隔离 |

## 九、面试价值点

| 维度 | 可以说什么 |
|:-----|:-----------|
| **架构能力** | 把 Agent 循环拆为编排层 + 决策层（loop_orchestrator / turn_kernel），互不耦合 |
| **审查系统** | 两级审查（正则/AST 毫秒级阻断 + LLM 子 Agent 秒级分析），`force=True` 逃生口 + false_positive 自动学习 |
| **Docker 沙箱** | 用 Docker 做测试沙箱隔离，代替 git worktree，容器销毁即回滚 |
| **import map** | 纯 Python AST 建表，O(1) 查表，后台线程建表不阻塞主循环 |
| **模式切换** | off→strict 过渡时自动触发全面回补扫描，不遗漏已有代码 |
| **记忆系统** | BM25 + 6 维评分，三层作用域，自动分类和分层 |
| **会话系统** | 增量 delta 保存 + 文件级检查点回退，字段级脏标记 |
| **插拔设计** | review/ 目录存在即启用，删除即回归原版，try/except ImportError 兜底 |
| **对比实验** | off / loose / strict 三级模式，可跑数据对比 |
