## 一、中文简历版（300 字以内）

### SmartCode - AI 编码 Agent 框架

**项目地址**：https://github.com/jiangkang11134/minicode

**技术栈**：Python 3.11+、AST、Docker、OpenAI 兼容 API、BM25

**项目概述**：
从零设计并开发了一个终端 AI 编码 Agent 框架（类似 Claude Code），具备完整的 think/act/verify 循环引擎、多工具调度、分层上下文管理、四级压缩引擎、增量会话持久化与文件级回退、三层作用域四级层级的跨会话记忆系统、两级代码审查和 Docker 沙箱测试。全项目零运行时依赖。

**核心子系统**：

1. **Agent 循环引擎**：三阶段架构（Prelude→Recurrent Kernel→Coda），策略推导与执行分离，分别由 turn_kernel（决策引擎）和 loop_orchestrator（执行引擎）承担。内置空响应重试、模型回退链、上下文四级压缩（微→轻→激进→反应式）带熔断器保护、widening 拓宽机制。每步产生结构化诊断日志

2. **三级记忆系统**：基于 Atkinson-Shiffrin 模型设计了三层作用域（USER/PROJECT/LOCAL）和四级层级（WORKING→SHORT_TERM→LONG_TERM→ARCHIVAL）。搜索使用 BM25 + 子串匹配 + 领域 Jaccard + 标签命中 + 时效性衰减 + 使用频率六维评分，领域采用软混合不过滤。自动分类器（8 类关键词启发式，零 LLM 依赖）。自动升降级、双格式持久化（JSON+Markdown）、原子写入。每轮对话实时 query 注入。额外工作记忆层保护 4 类关键上下文不被压缩丢弃

3. **会话持久化与文件回退**：实现增量 delta 保存策略（每 N 次增量后全量合并），字段级脏标记避免冗余序列化。文件编辑前自动创建检查点（checkpoint），支持按步数或按 ID 回退，回退前生成反向安全快照

4. **两级代码审查系统**：完全插拔式设计（review/ 目录存在即启用，删除即回归）。宽松层（正则+AST，0.72ms/次）在写入前阻断密钥泄露和 SQL 注入；严格层在检测到安全路径、新文件或大规模改动时自动启动子 Agent 做跨文件影响分析。审查通过后自动调 Docker 沙箱验证。运行时支持 off/loose/strict 三档动态切换，off → strict 过渡时自动触发全项目回补扫描

5. **Docker 沙箱测试**：容器化测试执行器，项目只读挂载 + 变更文件拷贝到沙箱，`docker rm -f` 销毁即回滚。测试通过零模型开销，失败时 3 轮轻量 Agent 分析

6. **Import Map 符号索引**：纯 Python AST 实现的跨平台符号索引，O(1) 查表替代 grep，后台线程惰性建表

**成果数据**：
- 102 个 Python 文件零语法错误，~42K 行，零运行时依赖
- 审查系统：准确率 100%，拦截率 100%，平均检测 0.72ms
- 端到端 20 个编码任务（单文件 + 多轮迭代 + 跨文件）全部通过
- 550+ 测试用例覆盖核心链路

## 二、英文简历版（350 words）

### SmartCode - AI Coding Agent Framework

**URL**: https://github.com/jiangkang11134/minicode

**Tech Stack**: Python 3.11+, AST, Docker, OpenAI-compatible API, BM25

**Overview**:
Designed and built from scratch a terminal-based AI coding agent framework (similar to Claude Code), featuring a think/act/verify loop engine, multi-tool orchestration, layered context management, four-level context compaction, incremental session persistence with file-level checkpoint rollback, a three-scope four-tier memory system, two-tier code review, and Docker sandbox testing — all with zero runtime dependencies.

**Key Subsystems**:

1. **Agent Loop Engine**: Architected a three-phase execution pipeline (Prelude→Recurrent Kernel→Coda) with clean separation between the decision engine (turn_kernel) and the execution engine (loop_orchestrator). Built-in resilience features including empty-response retry, model fallback chains, four-level context compaction (micro→light→aggressive→reactive) with circuit breaker protection, and automatic widening on depth stalls. Every step produces structured diagnostic logs

2. **Three-Tier Memory System**: Three scopes (USER/PROJECT/LOCAL) and four tiers (WORKING→SHORT_TERM→LONG_TERM→ARCHIVAL, based on Atkinson-Shiffrin). Six-dimensional search scoring: BM25 + substring + domain Jaccard + tags + recency decay + usage frequency, with soft domain mixing. Auto-classifier (8-category keyword heuristics, no LLM). Auto tier migration, dual-format persistence (JSON+Markdown), atomic writes. Per-turn query-driven injection. Working memory layer (15-entry) protecting 4 key context types from compaction

3. **Session Persistence & File Rollback**: Implemented incremental delta-based save strategy with periodic full consolidation. File edits are automatically checkpointed before modification, supporting rollback by step count or checkpoint ID, with reverse safety snapshots for data integrity

4. **Two-Tier Code Review System**: Fully pluggable design — the review/ directory enables the system, deleting it reverts to the original with zero residue. The loose tier (regex + AST, 0.72ms per check) blocks hardcoded secrets and SQL injection pre-write. The strict tier auto-launches read-only sub-agents for cross-file impact analysis triggered by security paths, new files, or large diffs. On review pass, automatically triggers Docker sandbox tests. Supports runtime switching between off/loose/strict with automatic retroactive full-project scan on off→strict transitions

5. **Docker Sandbox Testing**: Containerized test executor with read-only project mount + writable sandbox for changes. Container destruction equals full rollback with zero host contamination. Zero LLM overhead on test pass, lightweight agent analysis on failure

6. **Import Map**: Pure Python AST-based cross-platform symbol index providing O(1) affected-file lookups, replacing traditional grep. Background-thread build with zero startup blocking

**Results**:
- 102 Python files, ~42K lines, zero runtime dependencies
- Review system: 100% precision, 100% recall, 0.72ms avg check time
- End-to-end: 20/20 coding tasks passed (single-file, multi-turn, cross-file)
- 550+ test cases covering all core subsystems

## 四、面试深挖版（应对追问）

### Q1: 三级记忆系统的六维评分具体怎么实现的？

搜索评分：`(bm25 + substring + tag) × 0.7 + domain_jaccard × 0.3 + log1p(freq) × 0.3 + 1/(1+age_hours/24) × 0.5`。六个维度分别是 BM25 词频、子串匹配、领域 Jaccard（软混合不过滤）、标签命中、时效性指数衰减、使用频率对数增益。

层级迁移：WORKING → 30 天 → SHORT_TERM → 90 天 → LONG_TERM → 180 天 → ARCHIVAL。高频（≥5 次、>7 天）反向升级。归档后重新访问 7 天内激活。

### Q2: 会话增量保存怎么保证一致性？

增量保存只追加新消息和新检查点，不修改已有数据。每 10 次增量后做一次全量合并，删除冗余 delta 文件。回退时先读全量基础文件，再按时间序应用 delta。全量保存前先计算内容哈希，如果跟上次全量哈希一致则跳过，避免无意义的写入。文件写入使用原子操作（先写 tmp 再 rename），崩溃不丢数据。

### Q3: Agent 循环的 widening 机制是什么？

当模型在"深度模式"下连续多步没有进展（相同工具重复调用、返回空响应等），决策引擎自动切换到"广度模式"——增加可用步骤数、放宽验证严格度、注入拓宽提示引导模型换思路。触发原因和证据会被记录到结构化日志。

### Q4: 审查系统的 force=true 逃生设计为什么必要？

因为宽松审查是确定性规则，会有误报。比如一个合法的测试用例里写了 `password = "test"`，正则可能误判为硬编码密码。`force=true` 允许用户绕过阻断同时将这条规则记录到 false_positive 缓存，下次碰到相同 pattern 不再拦截——这是"渐进式精确"的设计。

### Q5: 上下文四级压缩各层触发条件是什么？

- **微压缩**：每次模型调用前，清除超过 60 秒的旧 tool_result（只保留摘要）
- **轻压缩**：上下文使用率 >70% 时，合并历史消息、丢弃低优先级内容
- **激进压缩**：策略推导标记 should_compact_aggressively 时，用 LLM 总结历史对话
- **反应式压缩**：API 返回 prompt too long 错误时，触发紧急压缩后重试

每层有熔断器保护（连续失败 N 次后跳过），防止压缩本身成为故障点。
