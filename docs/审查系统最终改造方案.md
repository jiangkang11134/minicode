# SmartCode — 审查系统最终改造方案

## 一、设计思路

### 1.1 为什么需要审查系统

SmartCode 是一个 AI 编码 Agent，能自主写代码、改代码、删代码。但它缺少一个关键环节：**代码质量把关**。

原项目中虽然有一个 `code_review` 工具，但：
- 从不自动触发 —— 写完代码模型不会主动调它
- 即使调了，也是模型审查自己的代码 —— 自答自审
- 只看单个文件 —— 改了一个函数，不知道哪些调用方会受影响

### 1.2 核心设计理念

**理念一：两级审查，各司其职**

| 级别 | 谁来审 | 耗时 | 职责 |
|:----:|:------|:----:|------|
| 宽松审查 | 正则 + AST | 毫秒级 | 拦住低级错误（密钥泄露、SQL注入） |
| 严格审查 | 审查子 Agent | 秒级 | 跨文件分析兼容性影响 |

宽松审查是"安检门"，严格审查是"专家会诊"。两道关卡，互不替代。

**理念二：插拔式架构**

整个审查系统是一个独立目录 `review/`：
- 目录存在 → 审查系统自动接入
- 目录删除 → 完全回归原版，零残留
- 不需要改任何配置、删任何代码

**理念三：不依赖 LLM 做确定性判断**

- 密钥是否泄露？—— 正则判断，不是 LLM 判断
- 代码是否被引用？—— import map 查表，不是 LLM 判断
- LLM 只做 LLM 擅长的事 —— 评估兼容性风险、判断误报、汇总报告

> **技术风险说明**：
> - `_pre_review_content` 的唯一实现在 `hooks.py`，`code_review.py` 只做 re-export，不存在两份规则
> - 子 Agent 的 `messages` 是主 Agent 当前列表的浅拷贝，状态隔离；`max_turns=8` 天然限制 token 预算
> - import map：建表 O(n) 后台执行不阻塞启动；查表 O(1)；增量更新也放后台线程，钩子 2 立即返回
> - test agent 回滚使用 `git checkout HEAD <file>` 只回滚指定文件，不影响其他文件

### 2.3 三级模式对比（插拔式开关）

```bash
MINICODE_REVIEW_MODE=off|loose|strict
```

| | L0: off | L1: loose | L2: strict |
|:----|:--------|:----------|:-----------|
| import map 建表 | 不建 | 后台惰性建 | 后台惰性建 |
| 宽松审查（正则+AST） | 不审 | 阻断写入 | 阻断写入 |
| import map 增量更新 | 不更新 | 更新 | 更新 |
| 严格审查触发 | 不触发 | 不触发 | 安全路径/diff/历史/新人 |
| 审查子 Agent | 不启动 | 不启动 | 直接 task 启动 |
| 误报学习 | 无 | force=True + 缓存 | 同左 |
| 对比实验角色 | 对照组 | 实验组 1 | 实验组 2 |

    def on_file_written(self, file_path):
        """钩子 2：写后更新 import map + 判断严格审查。"""
        update_import_map_for_file(self.cwd, file_path)
        if REVIEW_MODE == "strict" and should_trigger_strict(...):
            # 直接调 task 启动审查子 Agent

    def on_turn_end(self, turn_state) -> list[dict]:
        """钩子 3：Coda 阶段注入本轮累积的审查发现 + 沉淀到 skill/记忆。"""
        findings = self._pending_findings
        self._pending_findings = []

        # 沉淀重要发现到 skill 和全局记忆（跨项目共享）

1. `test` agent 返回 `[TEST_RESULT: FAIL]` → 结构化失败报告注入消息列表，LLM 下一轮直接看到失败详情
2. 同步沉淀到 `review_memory`（`.mini-code-import-map/review-findings.json`），记录 rule_id 和失败原因
3. 同 rule_id 累计失败 ≥3 次 → 自动沉淀为 skill（`~/.opencode/skills/`），下次写代码时系统提示中自动出现，从根源上避免重复犯错
}

## 四、已知技术风险

### 4.1 `_pre_review_content` 唯一实现

**风险**：`hooks.py` 和 `code_review.py` 各维护一套 `_CRITICAL_PATTERNS` → 改了一处另一处不同步。

**方案**：只在 `hooks.py` 中实现，`code_review.py` 只做一行 re-export。所有规则只有一处定义，不存在两份。

### 4.2 子 Agent 递归调用主循环

**风险**：钩子 2 中 `tools.execute("task", ...)` 内部调用 `run_agent_turn()` —— 主 Agent 循环里再起子 Agent 循环。

**方案**：
- **状态隔离**：`task` 工具传给子 Agent 的 `messages` 是主 Agent 当前列表的浅拷贝，子 Agent 的修改不回溯到主 Agent
- **Token 预算**：子 Agent 有自己的上下文窗口（`max_turns=8` 天然限制），不消耗主 Agent 的 context window
- **嵌套深度**：`task` 工具内部不会再次调 `task`，不存在递归风险
- **权限**：`review` 类型只含只读工具，不含任何写工具；`test` 类型虽然有 `git` 工具但只操作沙盒 worktree

### 4.3 import map 性能

**风险**：设计文档混用了"建表 O(n)"和"查表 O(1)"。增量更新的引用搜索仍需全项目扫描。

**方案**：
- 首次建表（3-8 秒）→ 后台线程执行，不阻塞 agent 启动
- 增量更新 → 放入后台线程，钩子 2 立即返回，`on_turn_end` 时确保完成
- 查表 → `get_affected_files()` 是 O(1) 字典查询
- 建表是 O(n)，查表是 O(1)，文档已明确区分

### 4.4 test agent 回滚粒度

**风险**：`git checkout HEAD` 会回滚所有未提交文件，不只是当前文件。主 Agent 改了 3 个文件，第 2 个测试失败 → 全部回滚 → 不一致状态。

**方案**：使用 `git checkout HEAD <file>` 只回滚指定文件，不影响其他文件的改动。

### 4.5 写后建表阻塞主循环

**风险**：`update_import_map_for_file()` 是同步操作，大项目下每次写文件后等几秒才继续。

**方案**：增量更新放入后台线程执行，钩子 2 立即返回。`on_turn_end` 时等待线程完成。

## 五、关键设计决策

| 决策 | 选什么 | 为什么 |
|:----|:------|:-------|
| 审查系统放哪 | 独立 `review/` 目录 | 目录存在即启用，删除即回归原版，try/except 兜底不报错 |
| 宽松审查正则放哪 | `hooks.py` 唯一实现 | `code_review.py` 只做 re-export，不存在两份规则 |
| import map 建表时机 | 后台线程惰性建（3-8s） | agent 启动不被阻塞 |
| import map 增量更新 | 后台线程执行 | 写文件后立即返回，不阻塞主循环 |
| 引用搜索实现 | 纯 Python（不用 grep） | Windows/Linux 跨平台，无外部依赖 |
| 严格审查触发方式 | 直接 `tools.execute("task")` | 不依赖主 Agent 的 LLM 决定，100% 触发 |
| review agent 工具集 | 只有只读工具 | 不能改文件，安全隔离 |
| test agent 回滚 | `git checkout HEAD <file>` | 只回滚失败文件，不影响其他文件 |
| 审查结果处理 | 注入消息（不阻断） | 保持代理自主性 |
| 误报处理 | `force=True` + 内存缓存 `_fp_cache` | 用户可强制写入，同模式不再误报 |
| 失败记忆 | 沉淀到 review_memory + 高频变 skill | 避免 LLM 重复犯同样错误 |
| 对比实验 | 三级模式 `off/loose/strict` | 有数据支撑，面试能说"做了实验" |

## 八、面试话术

> "我给 SmartCode 设计了一套完全插拔的两级审查系统。整个 `review/` 目录存在即启用，删除即回归原版，`agent_loop_lite.py` 只加了 3 处钩子共 20 行，全部用 `try/except ImportError` 兜底。
>
> **宽松审查**在 write 前执行正则+AST 检查，毫秒级完成，密钥泄露直接阻断写入。有 `force=True` 逃生口和 false_positive 自动学习——同一个 pattern 误报一次后不再拦截。
>
> **严格审查**在改了 auth 安全路径、大规模 diff、历史问题率高的文件时自动触发。它不依赖主 Agent 的 LLM 决定，直接 `tools.execute("task")` 调审查子 Agent。子 Agent 只有只读工具，基于 import map（纯 Python 跨平台实现，首次启动时后台建表不阻塞）查受影响文件，在 Coda 阶段将审查报告注入消息，代理自主决定是否采纳。
>
> import map 记录每个符号被哪些文件引用，每次写文件后自动增量更新。审查时查表而非 grep 全项目，从 O(n) 降到 O(1)。
>
> 我做了三组对比实验：`off` vs `loose` vs `strict`。`
