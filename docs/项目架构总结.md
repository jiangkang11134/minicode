# SmartCode 项目架构总结

> 基于最终完成态的完整代码（v0.2.0，103 个 Python 文件，~44K 行）

---

## 一、项目定位

SmartCode 是一个终端 AI 编码 Agent（类似 Claude Code 的开源实现），核心是一个 think/act/verify 循环引擎，运行在终端中，通过自然语言交互完成编码任务。

**关键数字：**

| 维度 | 数据 |
|:-----|:------|
| 源码 | 103 个 .py 文件，~44K 行 |
| 运行时依赖 | **零**（纯 Python 标准库） |
| 模型适配 | OpenAI 兼容 API（DeepSeek / Qwen / Claude / 自定义中转站） |
| 入口 | `pip install -e .` → `minicode` 命令 |
| 测试 | 31 个测试文件，~11K 行 |
| 最近更新 | CI/CD 流水线、成本控制、运行报告 |

## 三、核心引擎：run_agent_turn() 三阶段循环

这是整个系统的**心脏**，所有子系统围绕它组织。

```
run_agent_turn(model, tools, messages, cwd, ...)
  │
  ├─ Prelude ───────────────────────────────────────────
  │  ① TurnRecurrentState 初始化（步数/错误/策略追踪）
  │  ② 意图解析 → TaskObject + TaskGraph
  │  ③ 上下文预检：token 量检查 + 微压缩
  │  ④ init_review_hooks()（审查系统延迟初始化）
  │
  ├─ Recurrent Kernel（while has_remaining_steps()）───
  │  │
  │  Step A：策略推导 ← turn_kernel.py
  │    derive_turn_step_policy(turn_state)
  │    → explore（探索）/ execute（执行）/ verify（验证）
  │    → widning 判断（卡住时自动拓宽）
  │    → 更新 stable_task_pack（稳定任务状态）
  │
  │  Step B：模型调用 ← model_caller.py
  │    _model_next()
  │    → 预判式上下文守卫（token 超限检查）
  │    → 调 LLM（OpenAI/Anthropic 适配器）
  │    → 异常处理：ConnectionError / TimeoutError / PromptTooLong
  │    → fallback 链：主模型崩了切备用模型
  │
  │  Step C：响应判断 ← turn_kernel.py
  │    decide_assistant_turn()
  │    → "progress" → 中间思考，追加消息继续
  │    → "retry" → 空响应重试
  │    → "fallback" → widen 或终止
  │    → "final" → 完成，返回结果
  │
  │  Step D：工具执行 ← tool_executor.py
  │    _execute_single_tool()
  │    → 写前钩子：宽松审查（0.7ms）+ 严格审查（~5s）
  │    → ThreadPoolExecutor 超时保护
  │    → 多工具并发调度（只读并行，写入串行）
  │    → 写后钩子：import map 增量更新（后台线程）
  │
  │  ← 处理工具结果 → 错误分类 + nudge → continue
  │
  └─ Coda ─────────────────────────────────────────────
      ① 审查结果注入消息列表 + 重要发现沉淀
      ② metrics_collector.end_turn()
      ③ build_turn_coda_summary()
      ④ finalize_work_chain_task()
```

### 关键设计：决策与执行分离

| 模块 | 角色 | 类比 |
|:-----|:-----|:------|
| `turn_kernel.py` | **决策引擎**——做判断 | 导航系统："右转，限速 60" |
| `loop_orchestrator.py` | **执行引擎**——做动作 | 司机：踩油门，打方向盘 |

`turn_kernel.py` 不涉及任何 LLM 调用或工具执行，它是纯状态机 + 决策树。这让决策逻辑可以独立测试。

## 四、审查系统（本项目的核心产出）

### 4.1 插拔式架构

```
review/
├── __init__.py         包入口
├── config.py           配置（off/loose/strict 三级模式 + 运行时切换）
├── hooks.py            3 个钩子 + 宽松审查唯一实现（~470 行）
├── mode_engine.py      严格审查触发（4 层判断）
├── memory.py           审查发现持久化（JSON）
└── promotion.py        发现→skill/记忆自动沉淀
```

设计原则：`review/` 目录存在即启用，删除即回归原版。`try/except ImportError` 兜底，零配置残留。

### 4.2 两级审查流程

```
on_before_write()
  │
  ├── 阶段1：宽松审查（正则+AST，0.7ms）
  │   ├── _CRITICAL_PATTERNS（hardcoded-secret / sqli / unsafe-eval）
  │   ├── AST 检查（bare-except）
  │   └── _MAJOR_PATTERNS（except: 正则兜底）
  │
  │  命中 critical？→ 阻断写入（force=true 可跳过，记 false_positive）
  │
  └── 阶段2：严格审查（strict 模式，~5s）
      │ 带熔断器保护（连续 3 次 API 失败自动跳过）
      │ 带超时保护（10s 超时降级到宽松审查）
      │
      ├── 收集阶段（纯代码，零 LLM）
      │   ├── 环境摘要
      │   ├── 变更代码
      │   ├── import map → 受影响文件 + 内容
      │   └── code_review 结果
      │
      └── 1 次 LLM 调用 → 按等级处理
          ├── [CRITICAL] 安全漏洞 → 阻断 + 报告修复方案
          ├── [MAJOR] 兼容性问题 → 阻断 + 报告冲突分析 + 适配器建议
          ├── [MINOR] 优化建议 → 放行 + 附带报告
          └── [PASS] 无问题 → 放行
```

### 4.3 三个钩子的嵌入位置

| 钩子 | 位置 | 触发 | 行为 |
|:-----|:-----|:------|:------|
| **钩子 1** | `tool_executor.py` → `_execute_single_tool()` 执行前 | write/edit/patch | 宽松审查+严格审查，阻断或放行 |
| **钩子 2** | `tool_executor.py` → 工具执行成功后 | write/edit/patch | import map 增量更新（后台线程）+ test agent 触发 |
| **钩子 3** | `loop_orchestrator.py` → Coda finally 块 | 回合结束 | 注入 _coda_findings + promote_findings() |

### 4.4 review agent vs test agent

| 维度 | review agent（严格审查） | test agent（沙箱测试） |
|:-----|:------------------------|:----------------------|
| 触发 | on_before_write 同步 | on_file_written 异步 |
| LLM 调用 | 1 次（收集上下文后 LLM 分析） | **零**（失败后用 3 轮分析） |
| 工具 | task(agent_type="review") | sandbox_test 直接调用 |
| 阻断 | ✅ 阻断写入 | ❌ 不阻断（仅报告） |
| 耗时 | ~5s | ~30-60s（Docker 启动） |

### 4.5 运行时模式切换

```
MINICODE_REVIEW_MODE=off      → 完全不审查
MINICODE_REVIEW_MODE=loose    → 只开宽松审查（正则+AST）
MINICODE_REVIEW_MODE=strict   → 全开（宽松 + 严格）

off → strict 切换：自动触发全面回补扫描，遍历所有 .py 文件审查
```


## 五、记忆系统

### 5.1 三层作用域 + 四级层级

```
三层作用域（按生命周期）:
  USER    → ~/.mini-code/memory/       跨项目永久  不进 VCS
  PROJECT → .mini-code-memory/         随项目共享  可提交 Git
  LOCAL   → .mini-code-memory-local/   本地私有    不提交

四级层级（按访问频率）:
  WORKING → 当前会话中  → SHORT_TERM（<7天）→ LONG_TERM（<30天）→ ARCHIVAL
```

### 5.2 搜索评分（六维）

```
final_score = (bm25 + substring + tag) × 0.7
            + domain_jaccard × 0.3
            + log1p(usage_count) × 0.3
            + 1/(1+age_hours/24) × 0.5

BM25         → 词频 + 逆文档频率
子串匹配     → 关键词部分匹配加分
领域 Jaccard → 软混合（不匹配不排除，只降权）
标签命中     → 精确 tag 命中
时效性       → 指数衰减，越新越靠前
使用频率     → 对数增益，防高频条目过度占优
```

### 5.3 自动分类 + 升降级

```
自动分类器（8 类关键词启发式，零 LLM）:
  architecture / code-pattern / testing / configuration
  workflow / security / performance / convention

自动升降级:
  usage_count >= 5 且 age > 7 天   → SHORT_TERM → LONG_TERM
  last_accessed > 30 天             → LONG_TERM → ARCHIVAL
  归档后被重新访问 < 7 天           → ARCHIVAL → SHORT_TERM
```

### 5.4 工作记忆（纯内存层）

```
WorkingMemoryTracker（15 条上限）:
  active_task     → 活跃任务上下文（防止被压缩）
  user_intent     → 用户意图
  key_decision    → 关键决策
  error_context   → 错误上下文

// ConversationContinuityManager
  task_start / decision_point / error_recovered / user_redirect
```

## 六、会话系统

```
SessionData:
  messages / transcript / checkpoints / permissions_summary / skills / ...

增量 delta 保存:
  每次保存只追加新消息（比全量快 10x）
  每 10 次增量后全量合并
  字段级脏标记（只序列化变更字段）

文件级检查点回退:
  每次 write_file/edit_file 前 → create_file_checkpoint()
  支持按步数或按 checkpoint_id 回退
  回退前生成反向安全快照

原子写入:
  tempfile → os.replace
  写入中断不损坏原文件
```

## 七、上下文管理（四级压缩）

```
Level 1：微压缩（0.1ms）
  每次模型调用前执行
  清除 >60s 的旧 tool_result，只留摘要

Level 2：轻压缩（<100ms）
  上下文使用率 >70% 时触发
  合并历史消息，丢弃低优先级

Level 3：激进压缩（1-3s）
  策略推导标记 should_compact_aggressively
  用 LLM 总结历史对话

Level 4：反应式压缩
  API 返回 prompt too long 时紧急触发
  熔断器保护（连续失败 3 次后跳过）

熔断器：CompactionCircuitBreaker
  consecutive_failures >= 3 → 熔断
  任意成功 → 重置
```

## 十一、架构设计决策记录

| 决策 | 选择 | 理由 |
|:-----|:------|:------|
| 决策/执行分离 | turn_kernel vs loop_orchestrator | 决策逻辑可独立测试，互不耦合 |
| 审查系统位置 | review/ 独立目录 | 删目录即卸载，try/except 兜底 |
| 审查顺序 | 写前同步 vs 写后异步 | 写前阻断保证代码不落地 |
| test agent | 直接调 sandbox_test | 零 LLM 调用开销 |
| review agent | 收集+1次LLM vs 5轮agent循环 | 1 次 vs 5 次，降本 80% |
| 熔断器 | 上下文压缩 + 严格审查双熔断 | 防止 API 不稳时阻塞主流程 |
| 记忆搜索 | 软混合（不过滤）vs 硬过滤 | 不过滤保证不遗漏 |
| 会话保存 | 增量 delta vs 全量 | 每 10 次增量后合并，IO 减少 10x |
| 成本控制 | API 调用前检查 | 超过上限立即停止，不浪费 |
| 模型适配 | OpenAI 兼容 API | 覆盖 DeepSeek/Qwen/Claude/中转站 |
